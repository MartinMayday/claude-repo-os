## Gap analysis — full session assessment

Reviewing every layer produced across this thread against the governing principles: accuracy, completeness, enforceability, reusability, and deployment readiness.

### Gap 1 — Hook schema not verified against current Claude Code version

**What was built:** hook scripts in `.claude/hooks/` referenced as `preEdit`, `preCommand`, `postCommand` in `settings.json`.
**Problem:** Claude Code's documented hook event names and argument-passing conventions are versioned and have changed across releases. The schema used here was based on available information at the time but was never confirmed against the exact installed version.
**Risk:** hooks silently do nothing if the event names or invocation shape do not match.
**Severity:** High

### Gap 2 — Hook argument passing is assumed not confirmed

**What was built:** hooks receive `$1` as target file or command string.
**Problem:** Claude Code may pass hook context differently, for example as stdin JSON rather than positional arguments, depending on version.
**Risk:** guards run but receive empty input and pass everything, making enforcement invisible.
**Severity:** High

### Gap 3 — No smoke test script

**What was built:** smoke test described in prose.
**Problem:** no actual `smoke-test.sh` exists that verifies the system is alive end to end.
**Risk:** a new installation has no quick way to confirm the OS is working before real work begins.
**Severity:** Medium

### Gap 4 — Council skill not integrated as a native Claude Code skill

**What was built:** Council policy, request, and decision artifacts as custom markdown files.
**Problem:** the user's uploaded Council skill was discussed but never wired into the repo OS as an actual `.claude/skills/` entry.
**Risk:** Council capability exists conceptually but has no executable trigger inside Claude Code.
**Severity:** Medium

### Gap 5 — Skills layer not included in bootstrap

**What was built:** subagents in `.claude/agents/`.
**Problem:** the earlier session work on `.claude/skills/` for `plan`, `implement`, `debug`, and `review-pr` was never included in the bootstrap script.
**Risk:** a significant portion of the early session output is missing from the deployed repo.
**Severity:** Medium

### Gap 6 — `HEARTBEAT.md` has no automated updater

**What was built:** `HEARTBEAT.md` template and `heartbeat-update.sh` discussed.
**Problem:** `heartbeat-update.sh` was designed but never included in the bootstrap script or committed to the repo.
**Risk:** runtime state is a manual process, which breaks continuity under real autonomous use.
**Severity:** Medium

### Gap 7 — `DONE_GATE.md` has no hook enforcement

**What was built:** `DONE_GATE.md` checklist and `done-gate.sh` hook discussed.
**Problem:** `done-gate.sh` was designed but never committed. The done gate exists as a document but cannot block premature completion automatically.
**Risk:** the most important behavioral guarantee in the system is unenforced.
**Severity:** High
